# Automaton Definition

This note describes the data contract consumed by `@rethinkhealth/hl7v2-automata` when executing HL7 message structure DFAs. The definition is a pure-data export generated by the profile compiler so runtime packages can share the same DFA without compiling code.

## Shape

```ts
type Definition = Readonly<{
  start: number;
  finals: ReadonlySet<number>;
  transitions: TransitionMap;
  alphabet?: ReadonlySet<string>;
  effects?: Readonly<Record<string, Effects>>;
}>;
```

- **`start`** – numeric identifier of the initial state. All runs begin here.
- **`finals`** – set of accepting states. The runner checks if the final state is in this set to determine if the input sequence is accepted.
- **`transitions`** – sparse adjacency table keyed by state id. Each row is a `Map<string, number>` mapping symbols (`MSH`, `PID`, …) to destination states.
- **`alphabet`** _(optional)_ – full symbol inventory as a `Set<string>`, useful for diagnostics or tooling. The runner never enforces it; behavior is derived strictly from `transitions`.
- **`effects`** _(optional)_ – transition side effects keyed by `${state}:${symbol}`. The automaton engine treats each payload as an `Effects` struct, and generators are free to omit the map entirely when no metadata is required.

### TransitionMap

```ts
type TransitionMap = ReadonlyMap<number, Map<string, number>>;
```

A `Map` where each key is a state id and each value is another `Map` from symbols to destination states. Missing entries indicate that no transitions are valid from that state (or for that symbol from that state).

### Effects

```ts
type Effects = Readonly<{
  groupsOpened: readonly string[];
  groupsClosed: readonly string[];
}>;
```

Higher-level consumers interpret the payload. For HL7 profiles we use the `open`/`close` group identifiers emitted by the compiler.

## Runtime API

### RunnerInput

`RunnerInput` accepts either a segment symbol (`'PID'`) or an object with a `symbol` plus an optional `index`. When the index is missing, the runner assigns one automatically, ensuring consistent event indices. See [automaton-input.spec.md](./automaton-input.spec.md) for detailed normalization rules.

```ts
type RunnerInput =
  | string
  | Readonly<{
      symbol: string;
      index?: number | null;
    }>;
```

### Runner

The `runner()` function creates a stateful `Runner` instance with an incremental API:

```ts
const automaton = runner(definition);

// Process symbols one at a time
const event = automaton.consume("MSH");
if (event.type === "step") {
  // Valid transition, optionally with effects
  if (event.effects) {
    console.log("Groups opened:", event.effects.groupsOpened);
  }
} else if (event.type === "invalid") {
  // Invalid symbol
  console.error(`Invalid: ${event.symbol}`);
  console.error(`Expected: ${event.expected.join(", ")}`);
}

// Query state via properties
console.log("Accepted:", automaton.accepted);
console.log("Expected:", automaton.expected);
```

The runner maintains explicit lifecycle states (Ready → Running → Failed/Complete):

- **`consume(symbol)`** processes one input symbol and returns a tagged event ('step' or 'invalid'). Once Failed or Complete, subsequent calls return invalid events.
- **`accepted`** (readonly property) - whether the input was accepted by the automaton.
- **`expected`** (readonly property) - symbols that could be consumed next from the current state.

Create a new runner instance for each validation run. Instances are lightweight.

### Event Types

```ts
type RunnerStepEvent = Readonly<{
  type: "step";
  effects?: Effects;
}>;

type RunnerInvalidEvent = Readonly<{
  type: "invalid";
  symbol: string;
  expected: readonly string[];
}>;
```

## Consumption Tips

- Treat definitions as immutable and share them across runner instances.
- When `effects` is omitted from step events, assume no group metadata is available.
- Use `consume()` in a loop or with visitor patterns for incremental processing.
- Query `accepted` and `expected` properties to check the automaton's state.
- Create a new runner instance for each validation run (instances are lightweight).
- Once Failed or Complete, the runner returns invalid events for subsequent calls.
- State IDs are internal implementation details and not exposed in the API.
