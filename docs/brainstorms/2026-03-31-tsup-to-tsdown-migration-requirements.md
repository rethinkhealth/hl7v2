# Migrate Build Toolchain from tsup to tsdown

## Problem Frame

The `hl7v2-profiles` package contains ~10,800 profile files (fields, datatypes, tables across 13 HL7v2 versions). Each file is a dynamic `import()` target. tsup (esbuild) produces one chunk per dynamic import with no way to control chunk granularity — resulting in 10,796 output files, a 67-second build, and 38 MB of JS output. This causes slow npm installs (inode overhead), slow builds, and unnecessary filesystem pressure at runtime.

tsup's underlying bundler (esbuild) has no chunk size controls. tsdown, built on Rolldown (Rust-based, VoidZero/Vite team, 1.0 RC Jan 2026), provides `output.codeSplitting` with `minSize`/`maxSize` controls that solve this at the build layer without changing source files or codegen.

Since 39 packages in the monorepo use tsup, the migration covers the entire monorepo to unify the build toolchain rather than maintaining two bundlers.

## Requirements

- R1. Replace tsup with tsdown as the JS bundler across all 39 packages
- R2. Configure `codeSplitting.minSize` / `maxSize` on `hl7v2-profiles` to reduce chunk count from ~10,800 to a reasonable number (target: 200-600 chunks)
- R3. All existing package exports, entry points, and ESM output format must be preserved
- R4. TypeScript declarations continue to be generated by tsc (tsdown is JS-only, same as tsup)
- R5. Build scripts in each `package.json` updated from `tsup` to `tsdown`
- R6. Root and package-level devDependencies updated (remove tsup, add tsdown)
- R7. All existing tests must pass without modification (the migration is build-only, no source changes)

## Success Criteria

- `pnpm build` succeeds across all 39 packages
- `pnpm test` passes with no regressions
- `hl7v2-profiles` dist contains significantly fewer files (target: <600 JS chunks)
- `hl7v2-profiles` build time improves (baseline: 67s with tsup)
- No changes to source `.ts` files — only build config and dependency changes

## Scope Boundaries

- No source file changes (profiles, manifests, etc. stay as-is)
- No cross-version deduplication or data format changes (separate initiative)
- No changes to tsc declaration generation
- codeSplitting config only needed on `hl7v2-profiles` — other packages have simple entry points that don't need chunk control

## Key Decisions

- **tsdown over raw Rolldown**: tsdown is the library-focused wrapper with tsup-compatible config, making migration minimal
- **Full monorepo migration**: Unify on one bundler rather than maintaining tsup + tsdown side by side
- **Chunk control at build layer**: Use Rolldown's `codeSplitting` options rather than changing codegen or source structure
- **minSize/maxSize approach**: Let the bundler decide optimal grouping within size bounds rather than manual chunk assignment

## Dependencies / Assumptions

- tsdown is stable enough for production use (1.0 RC since Jan 2026, tsup-compatible API)
- Rolldown's `codeSplitting.minSize`/`maxSize` works correctly with dynamic `import()` targets
- tsdown supports all tsup config options currently in use (entry, format, sourcemap, target, esbuildOptions if any)
